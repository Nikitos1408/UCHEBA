## 1. Предметная область

Разрабатываемый продукт — настольная информационная система для ЦППМСП «Здоровье», предназначенная для:

- авторизации администратора и разграничения доступа;
- ведения картотеки пациентов;
- регистрации приемов пациентов специалистами;
- просмотра журнала приемов;
- формирования аналитических и статистических отчетов в формате PDF.

Система работает с базой данных `SQL Server` (`testjob`) через `Entity Framework 6` (основная работа с данными) и частично через `ADO.NET` (`SqlConnection`) для генерации отчетов.

## 2. Подход к выполнению задания

Для моделирования использован `PlantUML` как текстовый инструмент UML. Подход включал три шага:

1. Выделение ключевых акторов, сущностей и бизнес-сценариев по существующему коду проекта (`MainWindow`, `MainInterface`, `Cart`, `NewCard`, `NewJoin`, `Journal`, `Entities`, `Reports`).
2. Построение набора диаграмм, покрывающих статическую и динамическую части системы.
3. Генерация программного кода на основе диаграммы классов (доменная модель C#).

## 3. Реализованные диаграммы

Мною разработаны следующие диаграмы:

-  диаграмма вариантов использования (Use Case);
-  диаграмма классов (Class Diagram);
-  диаграмма последовательности авторизации;
-  диаграмма состояний процесса входа;
-  диаграмма деятельности сценария создания приема.

Комплект диаграмм раскрывает:

- **статическую сущность** системы (структуру классов, атрибуты, связи);
- **динамическую сущность** системы (взаимодействия объектов, состояния, поток действий пользователя).

## 4. Подробное описание каждой диаграммы

### 4.1. Диаграмма вариантов использования

<img width="466" height="679" alt="диаг вариантов использования" src="https://github.com/user-attachments/assets/22a7a850-ef4b-470b-98f0-19240dd5dbe3" />

Диаграмма фиксирует внешние требования к системе и показывает, какие функции доступны пользователю.

**Варианты использования.**

- пользователь: `Администратор`;
- варианты использования: `Войти в систему`, `Просмотреть картотеку пациентов`, `Создать/редактировать карточку пациента`, `Создать запись о приеме`, `Просмотреть журнал приемов`, `Сформировать PDF-отчет`.

**Ключевые зависимости.**

- `Войти в систему` включает проверку роли пользователя;
- `Сформировать PDF-отчет` включает обращение к журналу/данным за период;
- `Создать запись о приеме` расширяет  сценарий работы с карточкой пациента.

**Практическая ценность.** Диаграмма задает границы системы и служит основой для формулировки функциональных требований в дипломе.

### 4.2. Диаграмма классов

<img width="1191" height="870" alt="диаг классов" src="https://github.com/user-attachments/assets/c8ce8fb5-e97b-4a6b-8da3-92a6aa8abeea" />

Диаграмма описывает статическую структуру приложения: основные классы интерфейса, контекст данных и сущности предметной области.

**Основные классы.**

- инфраструктура: `App`, `testjobEntities7`;
- сущности БД: `Users`, `Roles`, `C1_Карточки_пациентов`, `C2_Прием_пациентов`;
- UI-уровень: `MainWindow`, `MainInterface`, `Cart`, `NewCard`, `NewJoin`, `Journal`;
- отчеты: `Rep_base`.

**Ключевые связи.**

- `App` инициализирует общий контекст `testjobEntities7`;
- `MainWindow` использует `App.context` для авторизации;
- `Users` связан с `Roles` по полю роли;
- `C1_Карточки_пациентов` агрегирует набор записей `C2_Прием_пациентов`;
- `MainInterface` выполняет навигацию к функциональным окнам и запуск отчетов.

**Практическая ценность.** Диаграмма используется как архитектурная карта проекта и как источник для генерации доменной модели кода.

### 4.3. Диаграмма последовательности 

<img width="1043" height="533" alt="диаг авторизации" src="https://github.com/user-attachments/assets/c1082737-6705-4973-b818-d52be4d564d5" />

Диаграмма показывает пошаговое взаимодействие объектов при авторизации пользователя.

**Участники**

- `Администратор` (инициатор);
- `MainWindow` (интерфейс входа);
- `App.context / testjobEntities7` (слой доступа к данным);
- `SQL Server` (хранилище учетных данных);
- `MainInterface` (следующее окно при успешном входе).

**Логика**

1. Пользователь вводит логин и пароль и нажимает «Войти».
2. `MainWindow` запрашивает пользователя через `Users.FirstOrDefault(...)`.
3. Контекст выполняет SQL-запрос к таблице `Users`.
4. При успешной проверке и роли администратора открывается `MainInterface`.
5. При ошибке показывается сообщение «Неверный логин или пароль».

**Практическая ценность.** Диаграмма подтверждает корректность динамики вызовов и помогает локализовать ошибки авторизации.

### 4.4. Диаграмма состояний 

<img width="758" height="1164" alt="диаг входа" src="https://github.com/user-attachments/assets/9e6395e2-128a-4e56-8650-92a03ad67263" />

Диаграмма отражает жизненный цикл окна входа и переходы между состояниями.

**Состояния процесса.**

- `Старт` и открытие `Форма_входа`;
- `Проверка_БД`;
- `Ожидание_ввода`;
- `Проверка_учетных_данных`;
- `Проверка_роли`;
- `Доступ_разрешен` и переход в `Главное_меню`;
- ветки ошибок: `Ошибка_подключения`, `Ошибка_авторизации`.

**Ключевые переходы.**

- при недоступности БД приложение завершает работу;
- при неверных учетных данных система возвращается в режим повторного ввода;
- при успешной проверке роли выполняется вход в главное меню.

**Практическая ценность.** Диаграмма формализует обработку исключений и отрицательных сценариев, что важно для раздела качества ПО.

### 4.5. Диаграмма деятельности

<img width="509" height="803" alt="диагр деятельности" src="https://github.com/user-attachments/assets/afb8b2f5-810b-4372-a58b-c7f85b60862b" />

Диаграмма описывает бизнес-процесс создания приема пациента как поток действий.

**Этапы процесса.**

1. Открытие главного меню и переход к операции `Новый прием`.
2. Проверка наличия карточки пациента.
3. Ветвление:
   - если карточка есть — выбор пациента;
   - если нет — создание карточки в `NewCard`.
4. Заполнение данных приема и валидация обязательных полей.
5. Ветвление по результату валидации:
   - данные корректны — сохранение в БД и сообщение об успехе;
   - данные некорректны — сообщение об ошибке и возврат к вводу.

**Практическая ценность.** Диаграмма демонстрирует алгоритм бизнес-операции и является основой для описания пользовательского сценария в дипломе.
